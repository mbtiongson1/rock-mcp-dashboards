# Investigation: recurring Rock 401 bursts (2026-07-20)

**Status:** OPEN — observability shipped; root-cause fix deferred pending live evidence.

## Symptom

Bursts of HTTP 401 from Rock across many unrelated endpoints within a single MCP
session, seconds to ~2 min, ~11 bursts/day, self-recovering. ~2,800 of 5,000
sampled error events on 2026-07-20.

## What the code establishes (static analysis)

- rock-mcp forwards the MCP client's **own Auth0 access token verbatim** to Rock
  (`src/http/oauth.ts` → `src/rock/auth-strategy.ts`); **no refresh/retry** in the
  Rock path. Token lifetime = the Auth0 access-token TTL, controlled by the client.
- No token is cached inside rock-mcp; the only TTL cache is 15-min Redis
  person-*metadata* that excludes all authorization fields (recomputed live).
- The MCP transport is **stateless** (`sessionIdGenerator: undefined`, fresh server
  per POST), `maxDuration=60`, Node runtime. A Vercel timeout is 504, never 401.

## Ranked hypotheses

- **(a) Auth0 access-token expiry between client refreshes — STRONGEST.** Shared
  credential → all endpoints 401 at once; self-heals on the client's next refresh;
  ~11/day ≈ TTL boundaries; 2-min = a stale-token batch draining before refresh.
- **(c) Rock-side rejection / clock skew — plausible secondary.** Rock JWT-config /
  Auth0 search-key backfill, or Rock↔Auth0/Vercel clock skew.
- **(b) Vercel lifecycle — ruled out on mechanism.**

## Decision gate — capture ONE burst 401 on `/mcp`

| Observed | Conclusion | Next action |
|---|---|---|
| 401 with `WWW-Authenticate: Bearer error="invalid_token"` | (a) client token expired at rock-mcp's gate | Read Auth0 API access-token TTL for audience `https://rock-mcp.favor.church`; correlate bursts with `POST /oauth/token` refreshes |
| Client sees 403 "Rock rejected your access token (401)" **and** audit `errorCode: ROCK_TOKEN_REJECTED` / `[rock-user-resolver] Rock rejected forwarded token` in logs | (c) Rock rejected a token rock-mcp accepted | Audit Rock JWT config + Auth0 search-key backfill; compare Rock vs Auth0/Vercel clock |
| 504 / `FUNCTION_INVOCATION_TIMEOUT` | (b) misclassified symptom | Investigate function duration, not auth |

**Observability shipped in this PR:** a Rock-401 during resolution is now flagged
(`ResolvedRockUser.tokenRejectedByRock`), logged with the subject, audited as
`ROCK_TOKEN_REJECTED`, and returns a distinct client message — so the next burst is
diagnosable from logs alone. **Known gap:** a token that passes `GetCurrentPerson`
but 401s on the parallel role checks surfaces as an `AccessDenied` 403 without the
flag; if the gate resolves to (c)-clock-skew, instrument those catches too.
